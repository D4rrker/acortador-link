## Convención de Commits

Para mantener un historial limpio y estructurado, este proyecto utiliza el estándar de [Conventional Commits](https://www.conventionalcommits.org/). La estructura obligatoria es: `tipo: descripción corta en minúsculas y en imperativo`.

### Tipos permitidos

* **`feat`**: Añade una nueva funcionalidad o característica. *(Ejemplo: `feat: añadir panel de analíticas`)*

* **`fix`**: Soluciona un error o bug en el código. *(Ejemplo: `fix: corregir error al guardar zona horaria`)*

* **`refactor`**: Mejora la estructura del código sin cambiar su comportamiento. *(Ejemplo: `refactor: migrar formulario a useActionState`)*

* **`chore`**: Tareas de mantenimiento, configuración o actualización de paquetes. *(Ejemplo: `chore: actualizar dependencias de Next.js`)*

* **`docs`**: Modificaciones exclusivas en la documentación. *(Ejemplo: `docs: actualizar instrucciones de instalación`)*

* **`style`**: Cambios de formato (espacios, comillas) que no afectan la lógica. *(Ejemplo: `style: aplicar Prettier a los componentes`)*

* **`test`**: Añadir o corregir pruebas automatizadas. *(Ejemplo: `test: verificar cálculo de clics totales`)*

* **`assets`**: Añadir o actualizar archivos de imagen, iconos, logos o recursos multimedia. *(Ejemplo: `assets: añadir iconos de alta resolución y og-image`)*
